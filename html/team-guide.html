Team Setup Guide

How to design and deploy AI agent teams with clawctl. Real examples for different team sizes, industries, and use cases — from a solo founder's first bot to an enterprise fleet.

Core Concepts

Three things to understand. Everything else is optional.

ConceptWhat it isExample
Team A group of agents that share a workspace, skills, and task queue. Created with clawctl team create. research, support, acme-corp
Agent An individual AI running in a Docker container. Has its own auth, config, and port. Created with clawctl create team/name. research/sarah, support/triage
Channel A messaging platform connection. People talk to agents through channels. Added with clawctl channel add. Telegram, Discord, Slack, WhatsApp

Optional Concepts (power users)

ConceptWhen you need it
RolesManager agents delegate tasks to worker agents. For teams with automated task routing.
PolicyAdmin-enforced security rules. Auto-created on setup with secure defaults. Edit if you need custom limits.
Access controlMultiple humans administering the same server. Controls who can run which commands.
Shared resourcesSkills, workspace files, and hooks shared across agents in a team. Enabled by default in teams.
Start simple. You only need to think about teams, agents, and channels. Everything else is handled by defaults or can be added later.

Team Patterns

Choose a structure based on how your agents need to collaborate.

1. Solo Agent

One agent, one channel. The simplest setup. Good for personal assistants and single-purpose bots.

You ──> [Telegram] ──> agent

2. Flat Team

Multiple agents at the same level. Each has its own channel and specialty, but they share a workspace so they can see each other's files. No hierarchy.

You ──> [Telegram] ──> team/frontend ─┐ You ──> [Discord] ──> team/backend ├── shared workspace You ──> [Slack] ──> team/devops ─┘

3. Manager / Worker

One manager agent receives tasks from humans and delegates to specialized workers. Workers report back to the manager. Best for automated workflows.

You ──> [Telegram] ──> team/lead (manager) │ ┌─────────┼─────────┐ v v v team/code team/test team/docs (workers)

4. Multi-Team

Separate teams with isolated workspaces. Each team is independent. Good for agencies serving multiple clients or departments with different data.

client-a/ client-b/ ├── analyst ├── writer ├── researcher └── editor └── shared workspace └── shared workspace (isolated) (isolated)

Example 1: Solo Founder

A personal AI assistant for a founder who wants help with coding, writing, and research.

1 agent 5 minutes Solo Telegram

Who this is for

You're a solo founder or indie hacker. You want a personal AI you can message from your phone at any time. It should remember context across conversations and have access to your project files.

Setup

# One command does everything clawctl setup

Follow the prompts:

# Team name my-team # Agent name assistant # Auth: choose Codex (1) 1 # Channel: choose Telegram (1) 1 # Paste your bot token from @BotFather 7123456789:AAH... # Add another agent? No n

That's it. Message your bot on Telegram, approve the pairing code, and you're live.

What you get

my-team/assistantYour agent, running on port 18789
Telegram connectedDM pairing required (strangers can't message it)
Codex authUses OpenAI models via OAuth
Tool profile: codingCan read/write files, run commands in its workspace
Tip: Put your project files in the agent's workspace directory and it can reference them in conversations. Find the path with clawctl status my-team/assistant.

Example 2: Startup Engineering Team

Three specialized agents sharing a codebase. The whole team talks to them on Slack.

3 agents 10 minutes Flat team Slack

Who this is for

A 5–15 person engineering team. You want agents that your whole team can interact with on Slack — one for frontend questions, one for backend, one for infrastructure. They share the same codebase so they understand the full system.

Setup

# Create the team clawctl team create eng # Create three specialized agents clawctl create eng/frontend --auth=codex clawctl create eng/backend --auth=codex clawctl create eng/infra --auth=codex # Connect each to your Slack workspace # (You need one Slack app per agent — see channels-guide.html) clawctl channel add eng/frontend slack --bot-token=xoxb-FE... --app-token=xapp-FE... clawctl channel add eng/backend slack --bot-token=xoxb-BE... --app-token=xapp-BE... clawctl channel add eng/infra slack --bot-token=xoxb-IF... --app-token=xapp-IF... # Start all three clawctl start eng/frontend clawctl start eng/backend clawctl start eng/infra

Customize each agent

Give each agent different instructions by editing their config:

# Make the frontend agent focus on React/TypeScript clawctl config edit eng/frontend # Add to agents.defaults.instructions: # "You are a frontend specialist. Focus on React, TypeScript, CSS..." # Make infra focus on AWS/Terraform/Docker clawctl config edit eng/infra # "You are an infrastructure specialist. Focus on AWS, Terraform, Docker..."

Team workspace

All three agents share eng/shared/workspace/. Clone your repo there:

cd ~/.openclaw/eng/shared/workspace git clone git@github.com:your-org/your-repo.git

Now all agents can read the same codebase and reference each other's work.

Architecture

#frontend channel ──> eng/frontend ─┐ #backend channel ──> eng/backend ├── shared workspace (your repo) #infra channel ──> eng/infra ─┘ shared skills
Tip: Invite each bot to its own Slack channel (#frontend, #backend, #infra). Team members join the channels they care about.

Example 3: Customer Support

Agents on different platforms handling customer inquiries. One for Telegram, one for Discord, one for WhatsApp.

3 agents 15 minutes Flat team Telegram + Discord + WhatsApp

Who this is for

A company that provides support across multiple platforms. Customers message on Telegram, Discord, or WhatsApp and get AI-assisted responses. Agents share a knowledge base (docs, FAQs, runbooks).

Setup

# Create the team clawctl team create support # One agent per platform clawctl create support/telegram-bot --auth=codex --telegram=7123456789:AAH... clawctl create support/discord-bot --auth=codex --discord=MTIz... clawctl create support/whatsapp-bot --auth=codex # WhatsApp needs interactive QR setup clawctl channel add support/whatsapp-bot whatsapp # Start all clawctl start-all

Add your knowledge base

# Put docs in the shared workspace cp -r ~/company/docs ~/.openclaw/support/shared/workspace/docs cp ~/company/faq.md ~/.openclaw/support/shared/workspace/ # Add a shared skill for support responses cat > ~/.openclaw/support/shared/skills/support-agent.md <<'EOF' You are a customer support agent for [Company Name]. Always be helpful, empathetic, and professional. Reference the docs in your workspace for accurate answers. If you don't know the answer, say so and offer to escalate. Never share internal information or pricing that isn't public. EOF

Enable outbound messaging (carefully)

By default, agents can only reply — they cannot initiate conversations. If you want an agent to proactively message approved contacts:

# Enable outbound on Telegram, but only to approved contacts clawctl channel send support/telegram-bot telegram --enable clawctl channel allow support/telegram-bot telegram 123456789
Security note: Outbound messaging requires an allowFrom list. The agent can only message contacts you explicitly approve. This is enforced by policy.

Monitor

# System overview clawctl status # Live dashboard (auto-refreshing) clawctl dashboard # Recent activity across all agents clawctl activity --since=4h

Example 4: Research Lab

A manager agent delegates research tasks to specialized workers. The manager coordinates; workers do the deep work.

4 agents 15 minutes Manager / Worker Telegram

Who this is for

A research team that wants to give a high-level instruction ("Investigate the feasibility of X") and have it automatically broken down and assigned to specialist agents.

Setup

# Create the team clawctl team create research # Create the manager (you talk to this one) clawctl create research/lead --role=manager --auth=codex --telegram=BOT_TOKEN # Create specialist workers clawctl create research/papers --role=worker --manager=lead --auth=codex clawctl create research/data --role=worker --manager=lead --auth=codex clawctl create research/code --role=worker --manager=lead --auth=codex # Start all clawctl start-all

How it works

You ──> [Telegram] ──> research/lead (manager) │ │ dispatches tasks │ ┌──────────┼──────────┐ v v v research/ research/ research/ papers data code (worker) (worker) (worker) │ │ │ └──────────┴──────────┘ │ shared workspace shared task queue

Using the task queue

The manager dispatches work. Workers claim and complete tasks.

# Manager creates a task (or you can create manually) clawctl task create research "Review papers on transformer efficiency" # See the queue clawctl task list research # A worker claims it clawctl task claim research TASK_ID --by=papers # When done clawctl task complete research TASK_ID
Tip: Workers write their results to the shared workspace. The manager can read those files when summarizing findings for you.

Example 5: Agency / Multi-Tenant

Separate isolated teams for each client. Data never crosses boundaries.

6+ agents 20 minutes Multi-team Mixed

Who this is for

An agency or consultancy deploying AI agents for multiple clients. Each client gets their own isolated team with their own data, credentials, and channels. No data leakage between clients.

Setup

# Client A: e-commerce company clawctl team create acme clawctl create acme/support --auth=codex --telegram=ACME_BOT_TOKEN clawctl create acme/analytics --auth=codex clawctl start acme/support clawctl start acme/analytics # Client B: SaaS startup clawctl team create widgetco clawctl create widgetco/assistant --auth=codex --discord=WIDGET_BOT_TOKEN clawctl create widgetco/developer --auth=codex --slack-bot=xoxb-WC... --slack-app=xapp-WC... clawctl start widgetco/assistant clawctl start widgetco/developer # Client C: law firm clawctl team create lawfirm clawctl create lawfirm/research --auth=codex clawctl start lawfirm/research

Isolation guarantees

BoundaryHow it's enforced
File systemEach team has its own shared/workspace/. Agents can only see their team's files.
DockerEach agent runs in its own container with cap_drop: ALL and no Docker socket access.
NetworkAgents are bound to loopback by default. They can't reach each other's ports.
CredentialsEach agent has its own instance.env with chmod 600. Tokens are per-agent.
ChannelsEach agent connects to its own bot. DM pairing ensures only approved users can chat.

Manage per-client with access control

# Grant your client liaison operator access to their agents only clawctl access grant alice operator # Then edit .access.json to scope alice to "acme/*" instances
Important: For true multi-tenant production, consider running separate clawctl instances (separate OPENCLAW_ROOT dirs) per client. This gives you OS-level isolation on top of container isolation.

Example 6: Enterprise Fleet

Full deployment with strict security, RBAC, audit logging, image pinning, and automated compliance.

8+ agents 30 minutes Multi-team + Manager/Worker Slack + Telegram

Who this is for

An organization deploying AI agents at scale. You need audit trails, role-based access, approved images, and policy enforcement. Multiple teams with different needs on the same server.

Phase 1: Secure foundation

# Initialize with secure defaults clawctl init # Tighten the policy clawctl config set policy requireSandbox true clawctl config set policy allowedImages '["openclaw:v2026.3.*"]' clawctl config set policy maxInstances 12 clawctl config set policy blockedChannels '["signal"]' # Set up access control clawctl access grant devops-lead admin clawctl access grant eng-team operator clawctl access grant interns user

Phase 2: Pin a tested image

# Pull a specific version clawctl image pull openclaw:v2026.3.25 # All new agents will use this image (enforced by policy)

Phase 3: Deploy teams

# Engineering team (manager + workers) clawctl team create eng clawctl create eng/lead --role=manager --image=openclaw:v2026.3.25 --auth=codex clawctl create eng/frontend --role=worker --manager=lead --image=openclaw:v2026.3.25 --auth=codex clawctl create eng/backend --role=worker --manager=lead --image=openclaw:v2026.3.25 --auth=codex clawctl create eng/qa --role=worker --manager=lead --image=openclaw:v2026.3.25 --auth=codex # Data team (flat) clawctl team create data clawctl create data/analyst --image=openclaw:v2026.3.25 --auth=codex clawctl create data/pipeline --image=openclaw:v2026.3.25 --auth=codex # Connect channels clawctl channel add eng/lead slack --bot-token=xoxb-ENG... --app-token=xapp-ENG... clawctl channel add data/analyst slack --bot-token=xoxb-DATA... --app-token=xapp-DATA... # Start everything clawctl start-all

Phase 4: Ongoing operations

# Daily status check clawctl status # Compliance validation clawctl policy validate # Review who did what clawctl access audit --since=24h # Full security audit clawctl audit # Upgrade all agents to a new image (with health-check rollback) clawctl upgrade --all --image=openclaw:v2026.4.1 # Back up critical agents clawctl backup eng/lead ~/backups/eng-lead-$(date +%Y%m%d).tar.gz

Fleet Operations

Once you have more than one or two agents, you need fleet-wide answers, not per-agent commands. These commands let you ask "what's broken?", "what drifted?", and "which agents are on which channels?" without shelling into containers or running per-instance loops.

Every command in this section is read-only unless explicitly stated. They compose existing surfaces (audit log, registry, Docker state) into one screen and emit copy-paste fix commands rather than executing anything themselves — the same pattern as policy validate.

1. clawctl errors — incident-triage umbrella

First command to run when something feels off. One screen, four sources:

  • Container state (running / restarting / exited + restart count)
  • Recent log errors per instance
  • Recent clawctl operations that returned error
  • Orphan Docker containers

Plus a Fix paths trailer with directive commands to address each finding.

# Default: 2-hour window clawctl errors # Wider window, JSON for dashboards clawctl errors --since=24h --json # Scope to one team clawctl errors --group=backend

2. clawctl drift — state consistency check

Detect any divergence between Docker reality and clawctl's view. Four dimensions in one pass:

Drift typeWhat it catches
Forward orphanDocker container named openclaw-* that isn't in the port registry. Usually a leftover from a test run or manual docker compose up.
Reverse orphanRegistry entry whose expected Docker container is missing.
Disk driftInstance directory on disk (has instance.env) that isn't in the registry.
Registry driftRegistry entry whose instance directory is gone.
clawctl drift clawctl drift --json | jq '.forward,.reverse'

3. clawctl orphans — find and clean stray containers

Where drift shows you everything, orphans is the focused tool for the most common drift class — a Docker container with the openclaw- prefix that's not in clawctl's registry. A restart-looping orphan can burn CPU and confuse incident triage.

# See what's orphaned clawctl orphans # Reverse view: registry entries with missing containers clawctl orphans --reverse # Remove a single orphan (asks for confirmation) clawctl orphans clean openclaw-stray-1 # Remove every orphan (asks once, then deletes) clawctl orphans clean --all --yes

4. clawctl logs --group -f — interleaved live tail

Tail every member of a group in one terminal with a per-member ANSI colour prefix. Ctrl-C exits cleanly. Without -f, a sequential dump with section headers. --grep is in-process and preserves order across members.

# Live multiplex tail of every backend agent clawctl logs --group=backend -f # Filtered live tail: only lines matching "401" clawctl logs --group=backend -f --grep=401 # Last 24h of errors, sequential dump clawctl logs --group=backend --since=24h --grep=error # Per-agent (same flags work) clawctl logs sarah -f --since=1h

5. clawctl auth verify + auth status --probe

Per-instance auth liveness, with honest confidence reporting. Tries three strategies in order:

  1. The auth-check endpoint (most confident).
  2. /readyz failing[] inspection.
  3. Log scan for auth errors in the last 5 minutes (a log-scan “ok” means no errors seen, not next call will succeed).
# Verify one agent — exits 0 only when verified clawctl auth verify sarah # Fleet auth inventory clawctl auth status # Same, but run verify per row and add a VERIFIED column clawctl auth status --probe # Re-run OAuth even when already verified (rare) clawctl auth sarah codex --force

6. clawctl channels — fleet channel matrix

Rows are agents, columns are channel types (telegram, discord, slack, signal, whatsapp). Cells show the dmPolicy when the channel is enabled, or — when not configured. Answers “which agents are on which channels?” without running channel status per agent.

# Fleet-wide matrix clawctl channels # Scope to one group clawctl channels --group=backend # JSON for dashboards: { columns: [...], rows: [{name, cells}] } clawctl channels --json
Pluralised vs singular. channels (plural) is the read-only fleet view. channel (singular) is the per-agent verb — add, remove, status, security, send, allow, deny.

When to reach for which command

SymptomStart here
"Something feels off in production"clawctl errors
"Docker shows containers I don't recognise"clawctl orphans
"An agent stopped responding on Telegram"clawctl auth verify <name> then clawctl logs <name> --since=10m
"Which agents are missing a channel?"clawctl channels
"A 401 is bouncing between agents"clawctl logs --group=<g> -f --grep=401
