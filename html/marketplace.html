01 — How it works

Three steps from URL to running agent.

A template is a JSON profile describing the desired state. claws apply reconciles your host to match — idempotent, schema-validated, secrets resolved at apply-time from env, files, or commands. Same template works in EC2 cloud-init and on a developer laptop.

1
Pick a template

Browse the marketplace or write your own profile.json. Templates declare runtime, policy, team, agents, channels, post-setup verifies. Schema-versioned and composable.

2
Supply secrets

Tokens and keys are referenced, never embedded. Drop them in /etc/claws/secrets/, env vars, or pull from your secret store via tokenFrom.command. Templates stay publishable.

3
Run claws apply

One command. Validates schema, resolves secrets, runs init + create + auth + channel + start, then audits and verifies. Exits 0 only on green. Re-runnable — converges to declared state.

Idempotent by design. Run the same apply twice — second time is a no-op. Edit your profile, re-apply — only the diff gets reconciled. Safe in cloud-init, safe in a cron, safe to hand to a junior engineer.
02 — Profile anatomy

One JSON file. Every primitive of claws, declaratively.

The schema (claws.ab0t.com/v1) maps one-to-one onto existing claws verbs. No new primitives — just orchestration.

// solo-telegram-coder/profile.json { "apiVersion": "claws.ab0t.com/v1", // schema version, validated "kind": "Profile", "metadata": { "name": "solo-telegram-coder", "version": "1.2.0", // semver for the template itself "description": "Personal coding bot on Telegram. Codex OAuth.", "author": "ab0t", "license": "MIT", "tags": ["solo", "telegram", "coding"] }, "runtime": { "name": "openclaw", // pick any registered runtime adapter "image": "openclaw:local" }, "policy": { "loopbackOnly": true, // bind gateway to 127.0.0.1 only "dmDefault": "pairing", // strangers must pair before DMs "outboundDefault": "off", "noNewPrivileges": true, "memoryLimit": "2G" }, "team": { "name": "default", // or your team name "shared": true }, "agents": [ { "name": "agent-1", "role": null, // "manager" | "worker" | null "auth": { "preferred": "codex", // or "apikey" "fallbackApiKey": { "provider": "openai", "fromEnv": "OPENAI_API_KEY" } }, "channels": [ { "type": "telegram", "tokenFrom": { "env": "TELEGRAM_BOT_TOKEN" }, // or { file } or { command } "dmPolicy": "pairing" } ], "tools": { "profile": "coding" } // scoped tool access } ], "postSetup": ["audit", "verify --all", "start --all"] // run after apply }

Every field is documented in the v1 schema reference. Templates can extend a base template and override specific keys (composition lands in v1.2).

05 — Multi-runtime

Templates work with any registered runtime.

claws is not OpenClaw-specific. The runtime adapter pattern (40-field Runtime struct, capability flags, custom registration via claws runtime init) means any agent harness with Docker + an HTTP gateway can be a first-class citizen.

openclaw ✓ Shipped

The original. Manager/worker topology via compose overrides, channel adapters for Telegram/Discord/Slack/Signal/WhatsApp, OAuth + API key auth.

auth channels workspace roles tasks policy
ab0t-sandbox v1.3

Per-agent ephemeral dev environment (Daytona-style). Code execution is sandboxed, filesystem is reset between sessions, no host bleed.

auth workspace code-exec channels (planned) roles (planned) ephemeral fs
custom ✓ Shipped

Bring your own. Define one JSON file mapping your harness's auth + start + readyz endpoints to claws's adapter contract, drop it in ~/.openclaw/runtimes/.

configurable user-defined JSON only composable
Want to see how custom runtimes work? Read docs/runtimes.md in the repo. The adapter is one file — no Go compilation required on your side.
06 — Same template, two journeys

Whatever runs in cloud-init also runs on your laptop.

The difference between the two modes is whether you ship secrets ahead of time or get prompted for them. Same engine, same idempotence guarantees, same verify step at the end.

Zero-touch (cloud) EC2 ready

For cloud-init, Dockerfile, k8s jobs, CI smoke tests. No prompts. Exits 0 only on green verify.

# EC2 user_data — fresh box → working agent in <90s #!/bin/bash set -e curl -fsSL https://raw.githubusercontent.com/ab0t-com/claws/main/scripts/install.sh | sh claws apply --template=solo-telegram-coder \ --secrets-dir=/etc/claws/secrets \ --yes --verify systemctl enable --now claws-agents.target

Pair with terraform/claws-ec2 for full IaC. AMI + secrets + template + verify all in one apply.

Interactive (laptop) Smart defaults

For developers trying claws for the first time. Prompts only for what only you can answer — the OAuth click and the bot token.

07 — Security model

Templates are publishable. Secrets are not.

A template you can put on the marketplace is a template that contains zero credentials. claws enforces this by design — every secret field is a reference, never a value.

🔐
Secrets resolved at apply-time

Profile fields like tokenFrom.env, tokenFrom.file, and tokenFrom.command are resolved when claws apply runs — never serialized into state. Missing secret = abort before mutating.

📋
Schema validation first

Every profile is validated against the embedded JSON schema before any state is touched. Unknown fields, missing required fields, and bad types fail loud with line-and-pointer errors.

⚠️
Warnings + explicit consent

Templates that request elevated permissions (allowlist disabled, root user, outbound on) must declare warnings: in metadata. apply shows them and prompts unless --yes is passed.

🪶
Audit + verify built in

Every template can include postSetup: ["audit", "verify --all"]. The audit runs your declared policy against actual state; verify probes each agent's auth + readyz. No green, no exit-0.

📦
Image pinning

Templates pin runtime images by tag or digest. Upgrades are an explicit template version bump — never a silent latest-pull.

🔏
Signing (v2.0)

The marketplace will require cosign/sigstore signatures on verified templates starting in v2.0. Bring-your-own templates always supported; verification opt-in until then.

09 — Contribute

Built a useful claws config? Share it.

We curate community submissions and promote good ones to verified status. Templates are MIT-licensed by default. Submission is one PR.

1
Author your profile.json

Validate against the v1 schema. Use existing templates as starting points. Run claws diff --file=./profile.json to preview changes before applying.

2
Write README + secrets.example.env

Describe the use case, prereqs, and what each agent does. List every env var the template references (without values). This is the user contract.

3
Open a PR

Push to ab0t-com/claws-templates. CI validates the schema, dogfoods on a fresh EC2, then a maintainer reviews. Verified status earned after 3 successful community uses.

10 — Roadmap

Where templates are going.

Phased so each step is independently shippable. No vapor.

v1.0.x · shipped
claws core + install pipeline

The binary, the installer, the release artifacts, fleet observability (errors, drift, orphans, auth verify, channels), the runtime adapter pattern. Foundation for everything that follows.

v1.1.0 · next
claws quickstart + minimal profile system

claws apply --file=<profile.json>, claws apply --template=<name> for bundled templates, the v1 schema, 4 bundled templates (solo, solo-telegram-coder, team-research, team-support), claws diff. ~2 days of work, ~600 LOC.

v1.2.0
Marketplace + remote templates

claws apply --template=github:org/repo@ref, claws template install/list/search, more bundled templates (personal-assistant, agency-multi-tenant, enterprise), warnings + confirmation for elevated permissions, template composition via extends:.

v1.3.0
ab0t-sandbox runtime + devbox template

A second first-party runtime. Per-agent ephemeral dev environments with sandboxed code execution. Ships the devbox template once the sandbox image is GA.

v2.0.0
Template signing + real registry

cosign/sigstore-signed templates. A real registry (claws.run or similar) with semver resolution, ratings, and a CLI search experience. Only after v1.x validates the model.

11 — Questions you might have

FAQ.

If yours isn't here, open an issue.

Where do templates actually live?
Bundled templates ship inside the release tarball under templates/. Remote templates (v1.2+) are fetched from GitHub repos. Your own templates can live anywhere on disk and are referenced by path.
Are templates safe to share publicly?
Yes — by design. Every credential field is a reference (tokenFrom.env, fromFile, fromCommand), never an inline value. Templates contain no secrets.
What if I need something none of the templates do?
Fork the closest one. Override what you need. Drop it in any directory. Run claws apply --file=./mine.json. No marketplace approval required for personal use.
Can I use a non-OpenClaw runtime?
Yes — claws ships with the runtime adapter pattern. Today: OpenClaw + user-defined custom runtimes. v1.3 adds ab0t-sandbox. Templates just declare runtime: { name: "..." }.
Is this Helm for AI agents?
Mental-model-wise, yes. Declarative manifests, idempotent reconciliation, semver, composable. Scope is smaller — one host at a time, no cluster orchestration. Helm-style chart system is overkill until that changes.
What about secrets management at scale?
Templates support tokenFrom.command: ["vault", "kv", "get", ...], so any CLI-accessible secret store works. Vault, sops, age, AWS Secrets Manager, anything that returns a string on stdout.
Will there be a paid tier?
No SaaS. claws is MIT-licensed forever. Templates are free. Future paid offerings (if any) would be hosted runtimes (the ab0t-sandbox product) or commercial support — never the core or the template ecosystem.
How do I report a malicious template?
Open a GitHub Security Advisory at ab0t-com/claws/security/advisories or email security@ab0t.com. We pull verified status pending review.
