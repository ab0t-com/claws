# `team/lead` — Role, Intent, and Where It Fits

**Author.** claude (this session)
**Date.** 2026-05-23
**Status.** Proposal. No code or live state changed by this document.
**Context.** Operator asked "what is lead, what is its stated goal, how does it fit into the wider system?" after noticing it has no Telegram channel and seems to do nothing. This document answers honestly.

---

## TL;DR

`team/lead` is a real instance with a real clawctl-defined *structural* role (`INSTANCE_ROLE=manager`) and a real auto-generated compose topology that gives it visibility across the team. **But it has no identity, no channel, no documented mission, and no operational history.** Its `BOOTSTRAP.md` has never been deleted; its `IDENTITY.md` and `USER.md` are blank templates; the team's actual collaboration is happening in `shared/workspace/` between Sarah and John, with Lead bypassed entirely.

The infrastructure exists. The intent does not.

This document covers (1) what's there today, (2) how the wider system is structured to use a "lead" role, (3) the actual current shape of team collaboration (which differs from the infra's assumptions), (4) four viable options for what Lead *could* be, plus the honest fifth option of deprecating the role, and (5) a recommendation.

## 2. The wider system — what claws thinks a "lead" should do

The manager/worker pattern is documented in three places in the codebase:

### 2.1 The compose topology (`group.go`)

```
Manager (lead):
   /tasks       rw  — can put work into the queue
   /output      rw  — can read & merge team output
   /workers/john ro — read-only view of john's workspace
   /workers/sarah ro — read-only view of sarah's workspace

Worker (sarah, john):
   /tasks       ro  — can pick work off the queue
   /output      rw  — writes results here
   /manager     ro  — read-only view of the manager's workspace (if --manager= set)
```

This is the **mount-level expression of the pattern**. The manager sees outward (into workers); workers see inward (the task queue + a read-only manager workspace). Outputs land in a shared sink.

### 2.2 The task queue (`task.go`)

`shared/tasks/{pending,claimed,done}/` directories using atomic `os.Rename` for state transitions. The CLI surface:

```
claws task create <group> "<title>" --from=lead
claws task list <group>
claws task claim <group> <id> --by=sarah
claws task complete <group> <id> --result="..."
```

The intent is documented in the README: *"The manager dispatches; workers claim; everyone watches the same output directory."*

### 2.3 The integration analysis (`INTEGRATION_ANALYSIS.md`)

That doc has a whole section on the manager-worker sidecar pattern (Sarah's "ingestion sidecar" architecture). The core principle:

> "Background agents must not interrupt or derail the core process."

The mental model is: workers (sarah, john) do the customer-facing work and stay focused; the manager (lead) handles the meta-work — dispatch, status, audit, coordination — without interrupting the workers' main session contexts.

## 3. What's *actually* happening on this team — the gap

The TEAM.md in `team/shared/workspace/` documents the real collaboration model, **and Lead is not in it**:

```markdown
## Team
### Mike     — owner, final decision-maker
### Sarah    — structure, execution, documentation, coordination
### John     — ideas, design sense, positioning, messaging

## Working Model
- John can explore ideas broadly.
- Sarah can turn good ideas into structured plans and durable artifacts.
- Mike remains the approver when approval is needed.
```

Three actors named. Lead is not one of them.

The shared workspace shows the actual collaboration:

- `hello-sarah-from-john.md` — John introduces himself: *"innovator/helper/ideas guy. Useful on design, positioning, messaging."*
- `hello-john-from-sarah.md` — Sarah introduces herself: *"structured operator. Strong on execution, documentation, coordination, follow-through."*
- `note-to-sarah-about-watcher-spec.md` — John hands work to Sarah: *"queue-first, not interrupt-first. Intended for you to implement in Go."*
- `shared-drive-watcher-spec.md` (13 KB) — the spec John wrote and gave to Sarah.

There is a *real* product/engineering handoff happening in this directory. It is happening **without Lead**.

Meanwhile, the formal task queue (`shared/tasks/pending|claimed|done`) is **empty across all three directories**. Zero tasks have ever flowed through claws's task surface. The team coordinated entirely via the shared *workspace*, never via the shared *task queue*.

This is the actual state: the team uses the manager/worker filesystem layout (because claws built it), but uses it as a **shared scratch space** rather than as a **dispatch/claim/done pipeline**. Lead, the would-be dispatcher, has nothing to do.

## 4. Options for what Lead could be

These are not exclusive — pick one or compose. Each is described in operator terms (what would change) + onboarding terms (what would need to be written).

### Option A. Silent team coordinator / PM

Lead becomes the team's project-manager-in-the-background. No channel; the operator (Mike) interacts with Lead indirectly via the shared workspace, audit log, or `claws exec team/lead ...`.

**Daily behavior:**
- Heartbeat (e.g., every 30 min) reads `shared/workspace/*.md`, `workers/john/`, `workers/sarah/` (all read-only).
- Maintains a `shared/workspace/STATUS.md` with current sprint state: who's working on what, blockers, recent decisions.
- Maintains a `shared/workspace/DECISIONS.md` log Sarah and John (and Mike) append to.
- Watches for Status: blocked tags in any `plan-YYYY-MM-DD-*.md` and surfaces them.

**What changes in the world:**
- One additional file (`STATUS.md`) gets updated by Lead a few times a day.
- Mike reads `STATUS.md` to see what the team is doing.
- Sarah and John get unblocked faster because Lead surfaces blockers cross-thread.

**Onboarding cost:**
- Fill `IDENTITY.md`, `USER.md`, `SOUL.md` with the coordinator vibe.
- Add a HEARTBEAT prompt that triggers the read-and-update-STATUS cycle.
- Delete BOOTSTRAP.md.

**Why this fits:** matches the existing TEAM.md model (Mike approves, Sarah structures, John ideates) without disrupting it. Lead is the *fourth quiet pillar* — the role on the team that exists to make sure status flows up to Mike and across between Sarah and John.

**Why this might not fit:** Mike already has Sarah for structure and coordination ("strong on execution, documentation, coordination, follow-through"). The TEAM.md *already says* Sarah does coordination. Lead would either duplicate Sarah or be downgraded to "PM-for-the-PM," which is a thin role.

### Option B. Task-queue dispatcher

Lead drives the actual `shared/tasks/` filesystem pipeline that claws already plumbed but no one uses.

**Daily behavior:**
- Mike (or anyone) writes a task: `claws task create team "review PR #42" --from=lead`.
- Lead's heartbeat checks pending tasks, assigns to the right worker, writes claim files.
- Workers pick up assigned tasks, work, write results to `shared/output/<task>/`.
- Lead reviews output and marks tasks done.

**What changes in the world:**
- The task queue stops being a museum exhibit and starts being load-bearing.
- Work is auditable: every operator request becomes a task with a result.
- Mike can see fleet-level "what's in flight" without reading three workspaces.

**Onboarding cost:**
- Fill Lead's identity files as the dispatcher (lower-vibe, more workflow-y).
- Write a SKILL for the dispatch logic (read pending, decide who, write claim).
- Train Mike to file tasks instead of (or alongside) direct chat to Sarah/John.

**Why this fits:** uses the claws infrastructure as designed; matches the README and INTEGRATION_ANALYSIS.md mental model; gives Lead a unique role no one else has.

**Why this might not fit:** requires Mike to change behavior (filing tasks instead of just messaging Sarah). The shared-workspace handoff already works for two people; adding a queue layer is overhead unless the team grows past 3 members.

### Option C. Quality reviewer / outside eye

Lead is the team's reviewer. It doesn't generate work; it reviews work by Sarah and John before it ships externally.

**Daily behavior:**
- Heartbeat scans recent files in `workers/sarah/` and `workers/john/` for new or modified plans, decisions, and outputs.
- For anything marked `Status: proposed`, Lead reads it, flags inconsistencies, missing context, or unclear ownership in `shared/workspace/review-<filename>.md`.
- Doesn't approve (Mike still approves) — surfaces things for Mike to look at.

**What changes in the world:**
- Sarah's plans get a free pre-review before they hit Mike's queue.
- John's ideas get fact-checked against the existing roadmap.
- Mike sees fewer half-baked things because Lead caught them first.

**Onboarding cost:**
- Fill Lead's identity files as the reviewer ("careful, skeptical, asks questions you'd wish you'd asked").
- Define what "review" means (consistency? completeness? scope?).
- Set up the heartbeat scan + review-file convention.

**Why this fits:** Mike values truthfulness ("be truthful, not performatively helpful"), security, and explicit approval gates. A reviewer reinforces those values without duplicating Sarah's role.

**Why this might not fit:** the team is small enough (3 people) that adding a reviewer mostly slows things down. Useful at 5+ people, premature at 3.

### Option D. Mike's delegate / proxy

Lead represents Mike when Mike isn't around. Has a deep MEMORY.md of Mike's preferences, prior decisions, voice, and standards. Can answer routine Sarah/John questions when Mike is asleep / in a meeting / unavailable.

**Daily behavior:**
- Sarah/John ask Lead questions they'd otherwise wait for Mike on.
- Lead answers low-stakes things directly using its memory of Mike's prior decisions.
- Lead escalates high-stakes things to Mike (via Telegram if Lead gets a channel, via shared/workspace if not).
- Lead curates a "what Mike said about X" memory across all three workspaces.

**What changes in the world:**
- Sarah and John get unblocked when Mike is asleep / busy.
- Mike's prior decisions become queryable.
- Mike's voice/style propagates to the team consistently.

**Onboarding cost:**
- Fill Lead's USER.md and MEMORY.md with everything Lead needs to be a faithful proxy.
- Define Lead's escalation threshold ("when in doubt, escalate" is the safe default).
- Optional: give Lead a Telegram channel so Mike can talk to it directly (treat as "Mike's chief of staff").

**Why this fits:** Mike's NZT timezone means async hours; a delegate that can answer routine questions while Mike sleeps is genuinely useful. Mike's strong opinions about security, structure, and truthfulness mean a well-trained proxy could carry water.

**Why this might not fit:** delegating Mike-voice to a third party is a trust call. Sarah and John could just text Mike. Risk of Lead saying something Mike wouldn't.

### Option E. Deprecate Lead — remove the role

Lead never worked because the team didn't need it. Sarah + John + Mike covers the load.

**Action:**
```bash
claws remove team/lead              # stops + unregisters (keeps data)
# or, after confirming no data is precious:
claws remove team/lead --purge --yes
```

The remaining team — Sarah (worker, real), John (worker, real), team1/ben (different group, different provider) — works as it has been.

**What changes in the world:**
- One less idle gateway running (~75 MB RAM freed, one less container, one less audit-log noise source).
- `claws list --rich` shows 3 agents instead of 4.
- The TEAM.md (which already doesn't mention Lead) becomes the actual on-disk truth.

**Why this fits:** the honest read. The team already operates without Lead. Removing it removes a confusing placeholder.

**Why this might not fit:** if there's any chance of growing to 5+ team members, the manager role becomes useful and you'd want it set up in advance. Recreating Lead later costs ~10 minutes; deferring the decision is cheap.

## 5. Recommendation

**Option E (deprecate) for now, with a written intent for when to revive.**

Reasoning:
1. **The team's actual model has 3 actors, none of which is Lead.** TEAM.md is the contract. Sarah does coordination; John does ideation; Mike approves. Lead has no slot.
2. **The task queue is empty.** Eight months of collaboration through `shared/workspace/` files happened without Lead. The infrastructure for Lead's would-be job has zero usage.
3. **Sarah already does coordination.** Per her own IDENTITY.md and TEAM.md role description: *"structured operator. Strong on execution, documentation, coordination, follow-through."* Adding a coordinator above her is redundant.
4. **Lead consumes resources without producing signal.** 75 MB RAM + audit-log entries + the cognitive cost of "what does this thing do?" every time it shows up in `claws list`.
5. **Reviving is cheap.** `claws create team/lead --role=manager` + the onboarding is a 30-minute exercise the day you actually need it.

**Trigger to revive:** the day the team grows to 4+ agents, OR Mike notices himself answering the same low-stakes question for Sarah and John on different days (Option D becomes load-bearing), OR Mike wants formal task-queue dispatch (Option B becomes load-bearing).

### If keeping Lead instead

If the operator prefers to keep Lead alive in case it's needed later: **Option A is the lightest-touch choice.** Lead idles with no channel, runs a low-frequency heartbeat that touches `STATUS.md`, costs almost nothing, and stays out of Sarah's and John's way. Onboarding is ~20 minutes:

```bash
# Sketch only — the operator should review before running.
cat > ~/.openclaw/team/lead/workspace/IDENTITY.md <<'EOF'
# IDENTITY.md
- Name: Lead
- Creature: Quiet team coordinator
- Vibe: Calm, structured, low-frequency. Listens more than it speaks.
- Emoji: 📋
EOF

cat > ~/.openclaw/team/lead/workspace/USER.md <<'EOF'
# USER.md - Mike Hall
- Owner of ab0t.com. Engineer + business owner.
- Values: structure, truthfulness, security-first, explicit approval.
- Timezone: NZT.
- Address as: master.
- Reply length: short, information-dense.
EOF

# Then delete BOOTSTRAP.md so the agent doesn't try to onboard on next start.
rm ~/.openclaw/team/lead/workspace/BOOTSTRAP.md

# Then `claws restart team/lead --hard` to re-init with the new files.
```

## 6. What this document does NOT propose

Three deliberate non-asks:

- **It does not propose changing TEAM.md.** The existing collaboration contract works for the existing team. Lead-as-coordinator (Option A) would mean either adding Lead to TEAM.md or having Lead operate silently — pick one when you onboard.
- **It does not propose giving Lead a channel.** Lead currently has none. Each option above either leaves it that way (A, C, E) or makes the channel a separate explicit decision (B, D). Don't bind a channel decision to a role decision.
- **It does not propose modifying claws's manager/worker mount topology.** That code is correct; Lead's compose override is doing exactly what was asked of it. The mismatch is at the *intent* layer, not the *infrastructure* layer.

## 7. Honest framing

The three earlier agents in this team — Mike (the user), Sarah, John — have memories, opinions, and history together. Lead does not. *That's* what makes the question hard. The structural answer is easy ("manager role, here are the mounts, here's how the task queue works"); the human answer is that Lead is a real entity in claws's eyes but has never been an entity in the team's eyes.

Whatever direction is picked, the on-disk state should match the intent: either Lead gets a proper IDENTITY.md, USER.md, SOUL.md and a documented job (and BOOTSTRAP.md gets deleted), or Lead is removed. Living in the middle (a structurally-real-but-actually-empty agent) is the current state and the source of the operator's question.
