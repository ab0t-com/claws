# Tasklist — v1.6.0 contract alignment + fleet ops + UUIDs

**Ticket:** `tickets/contract-alignment-fleet-ops-2026-05-24/ticket.md`
**Filed:** 2026-05-24

**Ordering principle: strictly by user impact + value.** P0 tasks fix
features that *look* implemented but silently don't work — they are the
highest-leverage thing we can ship. P1 tasks give operators visibility
into state that already exists on disk. P2 adds stable identifiers and
documentation. P3 is the deferral list.

---

## P0 — Silent feature failures (ship first)

Fixing these takes templates from "promises and doesn't deliver" to
"actually works". Highest user impact: existing v1.5 users running
templates with cron/hooks/skills get features that actually fire.

## P1 — Fleet/team operator visibility (ship next)

The data is on disk. Operators currently need `find + cat + jq` to
see it. These commands make the existing data legible.

| ID | Task | Files | Estimate |
|---|---|---|---|
| **P1.1** | **`claws team show <team>`** — ASCII topology tree (manager → workers, peer arrows). `--json` for tooling. Reads-only. | new `team_show.go` | ~120 LOC |
| **P1.2** | **`claws agent show <name>`** — single-screen overview: id, team, role, manager, peers, workers, channels, auth, sandbox, tools, skills, hooks, cron, sidecars. | new `agent_show.go` | ~150 LOC |
| **P1.3** | **`claws cron list <agent>`** — reads `cron/jobs.json` + recent `cron/runs/*.jsonl`. Columns: name, schedule, last-run, next-run, last-status. | `cron.go` | ~80 LOC |
| **P1.4** | **`claws cron tail <agent>`** — streams `cron/runs/*.jsonl` like `logs -f`. | `cron.go` | ~40 LOC |
| **P1.5** | **`claws team task-graph <team>`** — pending/claimed/done counts + per-task one-liners. | `team.go` | ~80 LOC |
| **P1.6** | **`claws fleet doctor`** — combines `doctor` + `audit` + `drift` + `orphans` with sectioned output. Exit non-zero on any failure. | new `fleet.go` | ~60 LOC |

**Verification gate (P1 done = ?):** A non-technical operator can run
`claws team show research` and see the manager/worker tree clearly,
plus `claws cron list research/lead` to see the cron jobs and their
next-run times.

## Hard rules

- **No breaking changes** to v1 schema. New fields only.
- **No deletion** of old paths in v1.6. Just write the new ones.
  Deprecation + removal in v1.7.
- **No runtime changes required.** All paths/formats match what the
  *current* OpenClaw runtime image already does.
- **Open questions in the ticket** must get answers (or assumptions
  documented) before P0.1 implementation begins. Wrong-shape jobs.json
  is worse than no jobs.json.

## Open questions (mirrored from ticket)

1. Confirm the cron jobs.json schema. Field list observed:
   `id`, `summary`, `runAtMs`, `nextRunAtMs`, `jobId`, `action`, `status`,
   `deliveryStatus`, `durationMs`. Need: the EDITABLE shape (what we write)
   vs the EMITTED shape (what runs/*.jsonl contains).
2. Does the runtime really mount + scan `shared/hooks/`?
3. Skills layout under `shared/skills/` — per-skill subdir?
4. Events endpoint: exists? path? auth?
5. UUID in openclaw.json `meta.id` — does the runtime do anything with it,
   or is it just a passthrough?

These are best answered by the runtime author or by reading the
runtime source. Without answers, P0.1 implementation makes educated
guesses based on the live `~/.openclaw/team/sarah/cron/` shape.
