# `claws` profiles, templates & one-click — design intent

**Last updated:** 2026-05-24
**Status:** design — not yet implemented
**Supersedes:** the original `claws quickstart` mini-design (now §6)

## 1. Project intent

claws is **a generic AI-agent control plane**, not just a manager for OpenClaw containers. The current OpenClaw integration is *one* runtime adapter; the architecture (40-field `Runtime` struct, capability flags, custom-runtime registration via `claws runtime init`) deliberately supports future harnesses:

| Runtime | Status | Notes |
|---|---|---|
| **OpenClaw** | ✅ shipped | The original — manager/worker topology, channel adapters |
| **ab0t sandbox** (Daytona-like) | 🔮 planned | Per-agent ephemeral dev environments with sandboxed code execution |
| **Custom** (user-defined via `runtime init`) | ✅ shipped | Anyone can plug in their own runtime by writing one JSON file |

The product surface is therefore:

> *"On any Unix host, declaratively bring up one agent or a coordinated team of agents — across any supported runtime, with any combination of channels, auth methods, and security policies — using either a named template or a custom profile. Idempotent, auditable, MIT-licensed."*

This document covers how we get there.

## 2. Two UX modes

We support **two distinct user journeys**, and the same primitives should serve both.

### Mode A — Zero-touch (cloud-native deployment)

**Story:** *"Spin up an EC2 box, point cloud-init at our installer, and 90 seconds later there's a working agent on Telegram. No human in the loop."*

```bash
# In EC2 user_data / cloud-init / Dockerfile / k8s job:
set -e
curl -fsSL https://raw.githubusercontent.com/ab0t-com/claws/main/scripts/install.sh | sh
claws apply --template=solo-telegram-coder \
            --secrets-dir=/etc/claws/secrets \
            --yes --verify
systemctl enable --now claws-agents.target
```

Requirements:
- Exits 0 if everything came up healthy; non-zero with a clear reason otherwise.
- No prompts. Ever.
- Reads secrets from files or env, never inline in the profile.
- Always runs `claws audit` + `claws verify --all` at the end. Failures are fatal.
- Idempotent: re-running on a partially-set-up box converges to the desired state.

Use cases: customer trials, dev environment provisioning, fleet rollout, CI smoke tests, demo boxes.

### Mode B — Smart defaults + interactive

**Story:** *"A developer on their laptop wants to try claws. They type one command, get prompted for the 3 things only they can answer (channel choice, bot token, OAuth click), and have a running agent in 5 minutes."*

```bash
$ claws quickstart
==> Pick a template (or just press enter for "solo"):
    1. solo               — one agent, one channel  [default]
    2. team               — manager + 2 workers, shared workspace
    3. support            — public-facing, allowlist enforced
    4. devbox             — single agent in a sandbox (ab0t-sandbox runtime)
==> Team name? [default]
==> Agent name? [agent-1]
==> Which channel? telegram / discord / slack / signal / whatsapp / none
==> Paste your bot token: ******
==> Auth method? codex (OAuth) / apikey
==> [opens browser for OAuth]
==> ✓ default/agent-1 ready and authenticated
==> ✓ telegram connected — try messaging your bot
```

Requirements:
- Sensible defaults shown in brackets, [Enter] accepts them.
- Each prompt explains what it's for ("Slack needs both a Bot Token and an App Token — see channels-guide.html").
- Skips any step that's already configured (idempotent).
- Same underlying engine as Mode A — Mode B is "Mode A with prompts where flags are missing."

Use cases: first-time install on a developer laptop, learning claws, reproducible demo on a colleague's machine.

## 3. The profile format

Profiles are **declarative JSON** describing desired state. Same shape whether shipped in our marketplace, written by a user, or generated by interactive setup.

```json
{
  "apiVersion": "claws.ab0t.com/v1",
  "kind": "Profile",
  "metadata": {
    "name": "solo-telegram-coder",
    "version": "1.2.0",
    "description": "One coding-focused agent on Telegram. OAuth via Codex.",
    "author": "ab0t",
    "license": "MIT",
    "tags": ["solo", "telegram", "coding"]
  },
  "runtime": {
    "name": "openclaw",
    "image": "openclaw:local"
  },
  "policy": {
    "loopbackOnly": true,
    "dmDefault": "pairing",
    "outboundDefault": "off",
    "noNewPrivileges": true,
    "memoryLimit": "2G"
  },
  "team": {
    "name": "default",
    "shared": true
  },
  "agents": [
    {
      "name": "agent-1",
      "role": null,
      "auth": {
        "preferred": "codex",
        "fallbackApiKey": { "provider": "openai", "fromEnv": "OPENAI_API_KEY" }
      },
      "channels": [
        {
          "type": "telegram",
          "tokenFrom": { "env": "TELEGRAM_BOT_TOKEN" },
          "dmPolicy": "pairing"
        }
      ],
      "tools": { "profile": "coding" }
    }
  ],
  "postSetup": ["audit", "verify --all", "start --all"]
}
```

### Key rules

- **No secrets in the profile JSON.** Every token/key references `fromEnv`, `fromFile`, or `fromCommand` (vault, sops, etc.). Profiles are publishable; secrets stay local.
- **Schema-validated.** `claws apply` validates against the embedded JSON schema before touching state.
- **Versioned.** `apiVersion` lets us evolve the schema without breaking old profiles.
- **Composable.** A profile can `extends: another-profile@v1` and override fields.
- **Idempotent.** Applying the same profile twice → same state. Diffs are detectable via `claws diff --profile=foo.json`.

## 4. Candidate templates (shipped with v1)

These are the templates we'd ship in-tree under `templates/` and use as the marketplace seed.

| Template | Agents | Runtime | Channels | Use case |
|---|---|---|---|---|
| **`solo`** | 1 | openclaw | none (added later) | Bare minimum. The `claws quickstart` default. |
| **`solo-telegram-coder`** | 1 | openclaw | Telegram | Personal coding bot. Codex OAuth. |
| **`solo-discord`** | 1 | openclaw | Discord | Personal assistant on Discord. |
| **`personal-assistant`** | 1 | openclaw | Telegram + memory enabled | Multi-channel, long-running personal AI |
| **`team-research`** | 3 (1 lead + 2 workers) | openclaw | Slack | Manager/worker task delegation. Shared workspace. |
| **`team-support`** | 2-3 | openclaw | Multi (Telegram + Slack) | Public-facing. **Allowlist enforced.** Audit on. Tool profile = `support`. |
| **`agency-multi-tenant`** | N teams, isolated | openclaw | Per-tenant | Separate group per client. Strict policy enforcement. Each team has its own allowlist + audit log. |
| **`enterprise`** | configurable | openclaw | Slack + audit pipeline | Full RBAC + policy enforce + scheduled backups. Compliance-leaning defaults. |
| **`devbox`** | 1 | ab0t-sandbox (planned) | none | Single agent in an ephemeral dev sandbox. Code execution allowed. |

### Template structure on disk

```
templates/
├── README.md
├── _schema/
│   └── profile.v1.json          # JSON schema for validation
├── solo/
│   ├── profile.json
│   ├── README.md                # what it does, prereqs, env vars expected
│   └── secrets.example.env      # env var names the user must populate
├── solo-telegram-coder/
│   ├── profile.json
│   ├── README.md
│   └── secrets.example.env
└── ...
```

Bundled templates ship inside the release tarball. Custom templates live anywhere on disk and are referenced by path: `claws apply --file=./my-team.json`.

## 5. Secrets

Secrets are *referenced*, never stored, in profiles:

```json
"tokenFrom": { "env": "TELEGRAM_BOT_TOKEN" }
"tokenFrom": { "file": "/etc/claws/secrets/telegram.token" }
"tokenFrom": { "command": ["vault", "kv", "get", "-field=token", "secret/claws/telegram"] }
```

`claws apply` resolves all `*From` references before invoking the underlying primitives. If any required secret is missing, it aborts before mutating state and tells you exactly which env/file/command was missing.

For Mode A (EC2), the convention is:

```bash
/etc/claws/secrets/
├── openai.key                   # mode 600
├── telegram.token               # mode 600
└── slack.tokens                 # mode 600  (KEY=value per line)
```

`secrets.example.env` next to each template documents the required entries.

## 6. The marketplace concept

Templates are first-class artifacts that can be:

1. **Bundled** — shipped in the release tarball under `templates/` (the curated set above).
2. **GitHub-hosted** — `claws apply --template=github:ab0t-com/templates/research-team@v1.2.0`
3. **Local files** — `claws apply --file=./my-team.json`
4. **Future: a real registry** — `claws.run/templates/personal-assistant` with semver, signing, ratings.

The marketplace is just "a list of well-known templates", served initially from the README of a `ab0t-com/claws-templates` repo. No infrastructure needed for v1.x.

### Trust model for marketplace templates

- Templates are **declarative**. They don't ship executable code — they reference existing claws primitives and existing Docker images.
- The runtime image (`openclaw:local`, `ab0t-sandbox:v1`) is still pinned and verifiable separately.
- A template that asks for unusual permissions (allowlist disabled, outbound on, root user) **must declare it in `warnings:`** and `claws apply` shows a confirmation prompt unless `--yes` is passed.
- Future: template signatures via cosign/sigstore.

## 7. What our existing system already allows

Cross-referencing the profile fields above with primitives we have today:

| Profile field | Existing primitive | Status |
|---|---|---|
| `runtime.name` | `claws runtime` (init, list, import, export) | ✅ shipped |
| `policy.*` | `claws policy init/validate/enforce` | ✅ shipped |
| `team.*` | `claws team create` / `claws group create` | ✅ shipped |
| `agents[].name` | `claws create <group>/<name>` | ✅ shipped |
| `agents[].role` | `--role=manager\|worker` | ✅ shipped |
| `agents[].auth` | `claws auth <name> codex \| apikey` | ✅ shipped |
| `agents[].channels` | `claws channel add` | ✅ shipped |
| `agents[].tools.profile` | `claws config set ... tools.profile` | ✅ shipped |
| `postSetup: ["verify --all"]` | `claws auth status --probe` (already implemented as part of fleet observability) | ✅ shipped |
| `postSetup: ["audit"]` | `claws audit` / `scripts/security-audit.sh` | ✅ shipped |

**So profile-apply is pure orchestration over existing verbs.** No new primitives needed for v1.1.

The sandbox runtime is a separate workstream that produces a new `ab0t-sandbox` Runtime adapter (a JSON file). Once that exists, `runtime: { name: "ab0t-sandbox" }` profiles "just work" via the existing adapter pattern.

## 8. Implementation roadmap

Staged so each phase is independently shippable.

### v1.0.x — fix-only (this week)

- ✅ Fix docker-compose.yml lookup (XDG data dir) — done, awaiting release.
- ❌ Nothing else in v1.0.x. Quickstart and profiles ship in v1.1.

### v1.1.0 — `claws quickstart` + minimal profile system

**Adds:**
- `claws quickstart [template]` — Mode B (interactive with smart defaults). Default template = `solo`. Idempotent.
- `claws apply --file=<profile.json> [--yes] [--verify]` — Mode A (declarative, non-interactive).
- `claws apply --template=<name>` — uses bundled templates from the release tarball.
- Profile schema v1 + JSON schema validation.
- Templates bundled: `solo`, `solo-telegram-coder`, `team-research`, `team-support`.
- `secrets.example.env` convention.
- `claws diff --file=<profile.json>` — shows what would change without applying.

**Doesn't add:**
- GitHub-hosted templates (use --file= for now).
- Marketplace UI.
- Template signing.
- Sandbox runtime.

**Effort estimate:** ~600 LOC across `apply.go` (new), `template.go` (new), `setup.go` (refactor), `runtime.go` (no change), embedded templates dir. ~2 days.

### v1.2.0 — marketplace + remote templates

**Adds:**
- `claws apply --template=github:org/repo/path@ref` — fetches from GitHub.
- `claws template install <ref>` — caches a template locally.
- `claws template list` — lists installed templates.
- `claws template search <query>` — searches the curated index (a README in `ab0t-com/claws-templates`).
- More bundled templates: `personal-assistant`, `agency-multi-tenant`, `enterprise`.
- Warnings declaration + confirmation prompt for templates requesting elevated permissions.

**Effort estimate:** ~400 LOC. ~1.5 days.

### v1.3.0 — `ab0t-sandbox` runtime adapter

**Adds:**
- `ab0t-sandbox` Runtime definition shipped in-tree.
- `devbox` template using the sandbox runtime.
- Documentation for the adapter contract (which fields a custom Runtime must populate).

**Effort estimate:** depends on the sandbox product — the adapter itself is ~1 day once the sandbox image API is stable.

### v2.0.0 — template signing + registry

**Adds:**
- cosign-signed templates.
- A real template registry (claws.run or similar) with semver + ratings.
- `claws apply --template=foo@^1.2` semver resolution.

Long horizon — only after v1.x validates the model.

## 9. Open decisions

1. **Profile format: JSON or YAML?** Existing config is JSON, so JSON for consistency. YAML support could be added later as syntactic sugar. **Recommend JSON.**
2. **Single profile or composed?** `extends:` adds complexity. **Recommend: single only in v1.1, add composition in v1.2 if real demand.**
3. **`claws quickstart` default template name?** `solo` (Recommend) or `default` or `hello`?
4. **`claws apply` vs `claws install` as the Mode-A verb?** `apply` matches the kubectl mental model (declarative). `install` is more familiar but implies one-shot. **Recommend `apply`.**
5. **Where do bundled templates live in the release tarball?** `templates/` next to `html/` and `docs/`. Discovered via the same XDG data dir lookup.
6. **Template metadata in marketplace index — JSON or just README?** README first (lowest friction), JSON index when we have >20 templates.
7. **Naming: "profile" vs "template" vs "manifest" vs "stack"?** Lean toward **"template"** for the user-facing word (familiar to Terraform/Helm users); the JSON document itself is technically a "profile" (instantiation of a template). Or pick one and stick with it.
8. **Should the sandbox runtime be part of this repo or a separate `ab0t-sandbox` repo with its Runtime JSON?** Separate repo — keeps claws's MIT scope narrow. Reference the runtime JSON from the marketplace.

## 10. Anti-goals (what we explicitly are NOT building)

- A web UI for claws. The CLI + JSON profiles are the surface.
- A SaaS control plane. Claws runs on your hosts; you own the data and the lifecycle.
- A replacement for Kubernetes/Nomad/Terraform. Claws is one level above — it orchestrates agent runtimes; another tool orchestrates claws.
- A package manager for skills/tools. Skills live in the runtime image; templates compose existing skills, they don't ship new ones.
- Sandboxing or sandboxing-as-a-service. The `ab0t-sandbox` runtime is a separate product; claws integrates it as an adapter, not as a feature.
